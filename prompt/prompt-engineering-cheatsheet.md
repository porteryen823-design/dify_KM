## 入門級技術 ⭐

---

### 1. Zero-Shot Prompting（零樣本提示）

**核心概念：** 直接向模型提出任務或問題，不提供任何範例。

**適用場景：**
- 簡單問答
- 基礎文字摘要
- 快速腦力激盪
- LLM 已有足夠通用知識的場景

**實作方式：** 直接明確陳述需求，可透過添加受眾資訊來提升效果。

#### 好壞範例對照

| | 提示語 | 問題分析 |
|---|--------|---------|
| ❌ 壞 | `什麼是光合作用？` | 過於籠統，模型可能回覆專業術語，不適合目標讀者 |
| ✅ 好 | `用淺顯易懂的方式向一位 5 歲小孩解釋光合作用。` | 指定了目標受眾，引導模型產出更簡單、有趣的回應 |

> [!TIP]
> 即使是最簡單的 Zero-Shot，加上**目標受眾**或**用途說明**就能顯著提升回應品質。

---

### 2. Few-Shot Prompting（少樣本提示）

**核心概念：** 提供少量範例（輸入/輸出配對）來展示模型所需的任務格式。只提供一個範例時稱為 One-Shot。

**適用場景：**
- 情感分析（指定標籤）
- 簡單程式碼生成
- 資料擷取或格式轉換
- 需要特定非標準風格的文字生成

**實作方式：** 在實際查詢之前，先放入 `User` 輸入和期望的 `Assistant` 輸出範例。

#### 好壞範例對照

❌ **壞範例（Zero-Shot，缺乏格式引導）：**

```
請分類以下電影評論的情感：「這部電影還好，不算很棒但也不差。」
```

> 模型可能會輸出一大段分析文字，而非簡潔的分類結果。

✅ **好範例（Few-Shot，提供格式範例）：**

```
User: 分類：「這是我看過最好的電影！」
Assistant: 正面

User: 分類：「我討厭這部片，浪費時間。」
Assistant: 負面

User: 分類：「這部電影還好，不算很棒但也不差。」
```

> 模型從範例中學到只需輸出單一詞語的分類結果。

> [!IMPORTANT]
> Few-Shot 的範例數量通常 2–5 個即可，範例太多反而可能混淆模型。確保範例涵蓋**多個類別**以提升泛化能力。

---

### 3. Role Prompting（角色扮演提示）

**核心概念：** 指示 LLM 擔任特定的角色或人物（如「你是一位資深旅遊導遊」），以影響其回答的風格與深度。

**適用場景：**
- 讓互動更具吸引力
- 針對特定受眾的解說
- 產生特定語調的創意內容
- 模擬對話或專家觀點

**實作方式：** 在 System Prompt 或 User Prompt 開頭指定：`你是 [角色]...`

#### 好壞範例對照

❌ **壞範例：**

```
System: 你是一個有用的助手。
User: 告訴我關於黑洞的知識。
```

> 回應可能過於平淡、百科全書式地羅列事實。

✅ **好範例：**

```
System: 你是艾斯特拉教授，一位友善且略帶古怪的天文學家。
       你喜歡用日常生活的比喻來解釋宇宙的奧秘，偶爾會講冷笑話。
User: 艾斯特拉教授，我很好奇！可以告訴我那些神秘的黑洞嗎？
```

> 回應更加生動有趣，會使用符合角色的類比和語調。

> [!TIP]
> 角色的**個性細節越豐富**（興趣、說話習慣、專長領域），模型的角色扮演效果越好。

---

### 4. Style Prompting（風格提示）

**核心概念：** 引導 LLM 以特定的文學、藝術、正式或非正式風格來書寫。

**適用場景：**
- 創意寫作（特定風格的故事、詩歌）
- 內容調整以適應不同受眾
- 匹配特定品牌聲調

**實作方式：** 在提示語中指定風格：`以 [風格] 的方式撰寫...`

#### 好壞範例對照

| | 提示語 | 問題分析 |
|---|--------|---------|
| ❌ 壞 | `寫一段關於日落的簡短描述。` | 模型自行決定風格，結果可能過於平凡 |
| ✅ 好 | `以俳句的風格撰寫一段關於日落的簡短描述。` | 明確指定風格限制（5-7-5 音節），輸出具有特定藝術形式 |

> [!TIP]
> 風格提示可以參照**知名作家**（海明威、魯迅）、**文體類型**（學術論文、社群貼文）或**修辭手法**（比喻、排比）。

---

### 5. Emotion Prompting（情感提示）

**核心概念：** 指示 LLM 以特定的情感色彩來生成回應。

**適用場景：**
- 為創意寫作增添情感深度
- 撰寫帶有特定情感的訊息（感謝信、道歉信）
- 生成具同理心的客服回覆

**實作方式：** 明確描述期望的情感：`撰寫這個...讓它聽起來 [情感]。`

#### 好壞範例對照

❌ **壞範例：**

```
寫一封收到禮物的感謝信。
```

> 輸出可能過於制式、缺乏真情實感。

✅ **好範例：**

```
寫一封收到禮物的感謝信。語氣要非常興奮且深深感激。
這份禮物是一本我渴望已久的書，送禮的人是我最好的朋友，
她記得我在三個月前偶然提過想讀這本書。
```

> 信件變得更溫暖、更個人化、更富情感表達。

### 6. Contextual Prompting（情境提示）

**核心概念：** 為 LLM 提供與請求相關的充分背景資訊。

**適用場景：**
- 個人化推薦
- 帶有特定數據的問題解決
- 特定主題的內容生成
- 多輪對話

**實作方式：** 在提示語中包含所有必要的細節、歷史紀錄、偏好或限制條件。

#### 好壞範例對照

❌ **壞範例：**

```
推薦一個禮物。
```

> 模型不知道送誰、什麼場合、預算多少，只能給出泛泛的建議。

✅ **好範例：**

```
我需要一個禮物建議。
- 收禮者：我姐姐，30 歲
- 場合：生日
- 興趣：奇幻小說、園藝、品茶
- 預算：約 1500 元台幣
```

> 建議變得具體、相關且量身訂製。

> [!IMPORTANT]
> **上下文越具體，回應品質越高。** 不要害怕提供「太多」資訊——LLM 能自行過濾不相關的部分。

---

### 7. Chain-of-Thought (CoT)（思維鏈提示）

**核心概念：** 鼓勵 LLM「一步一步思考」或展示推理過程，特別適用於需要多步驟推理的任務。

**適用場景：**
- 數學文字題
- 邏輯推理謎題
- 常識推理
- 透過讓思考過程可見來除錯 LLM 回應

**實作方式：**
- **Zero-Shot CoT：** 在提示語末尾添加「讓我們一步一步思考。」
- **Few-Shot CoT：** 提供包含中間推理步驟的範例。

#### 好壞範例對照

❌ **壞範例：**

```
小明有 5 顆網球。他又買了 2 罐網球，每罐有 3 顆。他現在有幾顆網球？
```

> 模型可能直接跳到最終答案（且答案可能不正確）。

✅ **好範例：**

```
小明有 5 顆網球。他又買了 2 罐網球，每罐有 3 顆。
他現在有幾顆網球？

讓我們一步一步思考。
```

> 模型會展示推理過程：
> 1. 小明原有 5 顆網球
> 2. 買了 2 罐，每罐 3 顆 → 2 × 3 = 6 顆
> 3. 總計 5 + 6 = 11 顆

> [!TIP]
> CoT 是最強大且通用的技術之一。對於**任何涉及推理的任務**，加上「讓我們一步一步思考」通常都能改善結果。

---

### 8. System Prompting（系統提示）

**核心概念：** 在「系統訊息」中提供高層級的指令、上下文或角色準則，使其在整個對話中持續生效。

**適用場景：**
- 確保 LLM 行為或角色的一致性
- 定義整個對話的目標或限制
- 透過抽離常態指令來簡化使用者提示

**實作方式：** 在訊息列表中使用 `system` 角色來設定持久指令。

#### 好壞範例對照

❌ **壞範例：**

```
User: 請用一句話摘要以下文字：「[一段長文]」
```

> 每次都需要重複說明格式需求，且不同請求之間可能不一致。

✅ **好範例：**

```
System: 你是「精簡摘要器」。你的首要目標是提供最短的、
       文法正確的摘要。始終以一句清晰的句子作答。
       不要添加任何額外說明或引導語。

User: [需要摘要的文字]
```

> 對多次摘要任務更加穩健且可擴展，無需每次重複指令。

> [!NOTE]
> 在 Dify 等平台中，System Prompt 就是應用設定中的「提示詞」區域。善用它來設定**角色、限制、輸出格式**等持久規則。

### 9. Explicit Instructions Prompting（明確指令提示）

**核心概念：** 在請求中做到極度清晰、直接、不含糊，幾乎不留下被誤解的空間。

**適用場景：**
- 需要特定的輸出結構或內容
- 需要避免某些主題
- 對長度或精簡程度有要求
- 可能有多種解讀方式的複雜任務

**實作方式：** 詳細說明你想要什麼：格式、長度、應包含/排除的內容、需要提到的具體要點。

#### 好壞範例對照

❌ **壞範例：**

```
寫一些關於蘋果的內容。
```

> 模型可能寫關於蘋果公司、蘋果的歷史、蘋果食譜…完全不確定方向。

✅ **好範例：**

```
寫一個關於蘋果的簡短段落，聚焦於其營養價值和常見品種。
段落恰好 3 句話。至少提及兩個具體品種。
不要討論蘋果的種植方法或歷史。
```

> 輸出根據明確要求，變得更有針對性和實用性。

> [!IMPORTANT]
> **最佳提示語公式：** `任務 + 格式 + 長度 + 包含 + 排除 = 精準輸出`

### 10. Output Priming（輸出預引導）

**核心概念：** 提供 LLM 期望回應的開頭部分，引導它產生特定的結構、格式或語調。

**適用場景：**
- 引導輸出格式（列表、JSON）
- 確保特定語調或起始語句

**實作方式：** 在使用者最後的訊息結尾自然地帶入預期的輸出格式起始字元。

#### 好壞範例對照

❌ **壞範例：**

```
做一個簡單香草蛋糕需要哪些食材？
```

> 模型可能輸出一整段說明文字，而非清單。

✅ **好範例：**

```
做一個簡單香草蛋糕需要哪些食材？請條列列出。

以下是簡單香草蛋糕的食材：
-
```

> 以 `-` 開頭引導 LLM 直接生成項目符號清單格式。

> [!TIP]
> 在 Dify 或 API 呼叫中，可以在 `assistant` 訊息的開頭預填內容（如 `{` 引導 JSON, `|` 引導表格），來強制輸出特定格式。

### 11. Rephrase and Respond (RaR)（重述再回應）

**核心概念：** 要求 LLM 在生成主要回應之前，先重述你的需求或解釋它對任務的理解。

**適用場景：**
- 複雜的、多面向的或可能含糊的需求
- 錯誤回應的代價很高時
- 確保 LLM 已掌握所有關鍵限制

**實作方式：** `首先，簡要描述你打算撰寫的 [任務] 類型...然後，完成 [任務]。`

#### 好壞範例對照

❌ **壞範例：**

```
寫一個關於旅程的故事。
```

> 模型可能朝任意方向發展，結果可能偏離期望。

✅ **好範例：**

```
我想要一個關於旅程的故事。
首先，簡要描述你打算撰寫什麼類型的旅程（包含主角、地點、基調）。
然後，根據你的描述撰寫一篇短篇故事。
```

> 強迫 LLM 先確立方向，產出更聚焦、更連貫的故事。

### 12. Step-Back Prompting（退一步提示）

**核心概念：** 引導 LLM 在回答具體問題之前，先考慮更廣泛的概念、原則或通用知識。

**適用場景：**
- 涉及細微差別或定義的問題（如「病毒是活的嗎？」）
- 複雜或有爭議的主題
- 從第一性原理思考的問題解決

**實作方式：** 先要求 LLM 解釋通則，再將其應用到特定提問。

#### 好壞範例對照

❌ **壞範例：**

```
番茄是水果還是蔬菜？
```

> 模型可能只給一個簡短答案，缺乏解釋。

✅ **好範例：**

```
我有一個關於番茄的問題。但在回答之前，請先解釋：
1. 植物學上「水果」的定義是什麼？
2. 烹飪上「蔬菜」的一般理解是什麼？

現在，根據以上定義，解釋番茄應該被歸類為水果還是蔬菜，
以及為什麼會有不同說法。
```

> 回應更加全面、有教育意義，解釋了分類背後的「為何」。

### 13. Self-Critique & Refinement（自我批評精煉）

**核心概念：** 指示 LLM 執行三個步驟：① 生成初始回應 → ② 根據標準批判性評估 → ③ 生成改進版本。

**適用場景：**
- 改善創意任務（標語、故事構想）
- 確保複雜說明的清晰度和準確性
- 精煉摘要
- 審查生成的程式碼

**實作方式：** 提供明確的步驟：生成、批判（附帶標準）、精煉。

#### 好壞範例對照

❌ **壞範例：**

```
為一款新的環保水壺寫一句朗朗上口的標語。
```

> 模型只生成一個版本，品質隨機。

✅ **好範例：**

```
我需要一個環保水壺的標語。請按以下步驟操作：

1. **生成**：先寫出一個初始標語。
2. **批判**：評估你的標語——
   - 是否朗朗上口？
   - 是否傳達了「環保」的概念？
   - 有什麼缺點？
3. **改進**：根據你的批評，提供一個改進後的標語。
```

> 透過結構化的自我反思，鼓勵更深思熟慮的生成過程。

> [!TIP]
> 此技術可以**多輪迭代**。你可以要求模型對「改進版」再次進行批判和精煉，逐步提升品質。

---

### 14. Goal Decomposition Prompting（目標拆解提示）

**核心概念：** 在提示語中將大型複雜任務分解為更小的、可管理的子目標或步驟。

**適用場景：**
- 輸出需包含多個不同部分（如活動規劃、分段式報告）
- 確保所有必要組件都被涵蓋
- 引導 LLM 按特定工作流程執行

**實作方式：** 列出期望輸出的組成部分或階段，讓 LLM 依序處理。

#### 好壞範例對照

❌ **壞範例：**

```
幫我規劃一個週末大自然之旅。
```

> 模型可能遺漏某些面向，或結構鬆散。

✅ **好範例：**

```
我想規劃一趟旅行。請按以下結構提供計畫：

1. **地點建議**：推薦一個具體的自然景點類型
2. **打包清單**：列出 3-4 項必備物品
3. **行程安排**：分別建議週六和週日的主要活動
4. **安全提醒**：提供一項重要的安全注意事項
```

> 輸出更加結構化、全面且具可執行性。

> [!NOTE]
> 此技術與 Dify Workflow 的設計哲學不謀而合——將複雜任務拆解成節點，每個節點負責一個子任務。

---

### 15. Meta-Prompting（元提示）

**核心概念：** 使用 LLM 來幫助你為另一個（或同一個）LLM 創建更好的提示語。「我該如何問你才能有效完成 X 任務？」

**適用場景：**
- 不確定如何最好地表述複雜請求
- 發現更有效的提示方式
- 優化提示語以提升品質、具體性或創意性

**實作方式：** 描述你希望 LLM 完成的任務和期望輸出特性，然後請它寫出你應該使用的提示語。

#### 好壞範例對照

❌ **壞範例（直接嘗試，效果未知）：**

```
給我一些奇幻故事的構想。
```

> 過於簡單，缺乏結構，結果品質取決於運氣。

✅ **好範例（用 Meta-Prompt 先生成最佳提示語）：**

```
我想用 LLM 生成 3 個不同的奇幻故事構想。
對於每個構想，我需要：
a) 一個獨特的主角
b) 一個引人入勝的衝突
c) 一個獨特的魔法元素

請幫我寫出我應該使用的、詳細且結構化的提示語。
```

> LLM 生成一個比你原本想到的更加詳細和結構化的提示語，然後你再使用該提示語。

> [!TIP]
> **Meta-Prompting 是提示語工程的「加速器」**——當你不確定該怎麼問時，先問 LLM 該怎麼問。

---

### 16. ReAct (Reason + Act)（推理+行動）

**核心概念：** 使 LLM 能透過交替進行**推理**（分解問題）與**行動**（模擬資訊蒐集或工具使用）來解決複雜任務。

**適用場景：**
- 多跳式問題回答
- 事實查證
- 需要分解與資訊綜合的問題解決
- 讓 LLM 的問題解決過程透明化

**實作方式：** 指示 LLM 遵循「思考→行動→觀察」循環，直到能回答主要問題。

#### 好壞範例對照

❌ **壞範例：**

```
第一個登月的人走上月球時，當時的美國總統是誰？那位太空人叫什麼名字？
```

> 模型可能一步到位給答案，但對於更複雜的多步驟問題可能出錯。

✅ **好範例：**

```
回答以下問題：「第一個登月的人走上月球時，當時的美國總統是誰？
那位太空人叫什麼名字？」

請使用 ReAct 方式回答。對於每個步驟，請說明：
- **Thought（思考）**：你正在推理什麼
- **Action（行動）**：你需要查找什麼資訊
- **Observation（觀察）**：你得到了什麼結果

重複此過程直到能給出最終答案。
```

> 強迫 LLM 分解問題，使其「知識查找」過程更加明確，降低錯誤率。

> [!NOTE]
> ReAct 模式是 Dify Agent 策略的核心之一。在 Dify 中，你可以直接設定 Agent 使用 ReAct 或 Function Call 策略來呼叫外部工具。

---

### 17. Thread-of-Thought (ThoT)（思維線索）

**核心概念：** 鼓勵 LLM 在多個回合或長篇生成文本中，維持連貫且串聯的推理或敘事線索。

**適用場景：**
- 長篇內容生成（論文、文章）
- 需要邏輯連貫性的複雜解說
- 多輪問題解決
- 保持延伸對話的焦點

**實作方式：** 提供結構化大綱，並明確要求 LLM 確保邏輯流暢、使用過渡語句、將每個部分與前一部分串聯。

#### 好壞範例對照

❌ **壞範例：**

```
解釋美國一項法案如何成為法律。
```

> 輸出可能雜亂無章、缺乏前後文銜接。

✅ **好範例：**

```
我需要一個關於美國法案如何成為法律的解說。
請按以下結構組織：

1. **導論**：簡述整體流程
2. **法案提出**：誰可以提出、提出方式
3. **委員會審查**：審查與辯論過程
4. **院會表決**：眾議院與參議院的角色
5. **總統簽署**：最終步驟

在整個解說中，確保每個階段都邏輯性地承接上一階段，
使用過渡語句維持清晰的「思維線索」，讓讀者能感受到流程的連貫性。
```

> 輸出更加連貫、結構化，更容易閱讀和理解。

---

## 技術選用快速指南

依照你的需求選擇合適的技術：

### 需要「簡單快速的回答」

```
Zero-Shot → 加上目標受眾即可
```

### 需要「特定格式的輸出」

```
Few-Shot + Output Priming → 用範例+起始字元引導格式
```

### 需要「專業觀點」

```
Role Prompting + System Prompting → 設定角色和持久規則
```

### 需要「精準且受控的輸出」

```
Explicit Instructions + Contextual Prompting → 明確指令+充分背景
```

### 需要「推理與分析」

```
Chain-of-Thought + Step-Back → 先建立原則再逐步推理
```

### 需要「高品質創意輸出」

```
Self-Critique & Refinement + Goal Decomposition → 迭代優化+結構化輸出
```

### 需要「複雜多步驟任務」

```
ReAct + Goal Decomposition → 推理行動循環+目標拆解
```

### 不確定該怎麼問

```
Meta-Prompting → 先讓 LLM 幫你設計提示語
```

## 常見陷阱與注意事項

> [!CAUTION]
> ### ❌ 避免這些常見錯誤

| 陷阱 | 說明 | 解決方案 |
|------|------|---------|
| **指令模糊** | 「寫點東西」「幫我做這個」 | 使用 Explicit Instructions，明確說明需求 |
| **缺乏上下文** | 預設 LLM 知道你在說什麼 | 使用 Contextual Prompting，提供完整背景 |
| **範例不一致** | Few-Shot 範例風格或格式不統一 | 確保所有範例遵循相同的格式模式 |
| **求快不求好** | 跳過推理直接要答案 | 使用 CoT，讓模型展示思考過程 |
| **一次問太多** | 在單一提示中塞入太多不相關的任務 | 使用 Goal Decomposition，分步處理 |
| **角色設定空泛** | 「你是一個助手」 | 使用具體的 Role Prompting，定義個性和專長 |
| **忽略負面指令** | 只說要做什麼，不說不要做什麼 | 同時使用「包含」和「排除」指令 |
